왜 프레임워크 없이 직접 구현했는가

  • LangChain/ReAct의 "thought" 출력 → 2,000 에이전트에서 토큰 비용 7배 증가
  • 프레임워크 메모리 모듈이 "인상(impression)"과 "의견(opinion)" 구분 불가 — 연구 핵심 설계와 충돌
  • tool_loop.py 135줄 상태머신으로 모든 도구 호출이 DB에 기록 → 학술 재현성 확보

핵심 설계 결정

  • 자율 판단 원칙: 프롬프트에 행동 지시 없음 — 에이전트가 도구를 스스로 선택
  • 인상 축적 모델: 뉴스 소비 → 인상만 기록 → 투표 시점에만 의견 표출
  • 듀얼 LLM: GPT-5 mini + Gemini Flash로 모델 편향 분리 (RQ2)
  • Gemini Flash Lite: $0.10/1M으로 대규모 시뮬레이션 비용 최적화

핵심 구현

  • 내면 축적 메커니즘: 매 라운드 에이전트 반응을 2~3문장 "인상"으로 압축·누적. 의견 아닌 느낌을 축적하여 자기강화 루프 방지, 라운드당 ~100토큰으로 비용 효율적
  • 뉴스 파이프라인 v2: RSS 4개 매체(보수2+진보2) → Trafilatura 본문추출 → LLM 분류(Enum 강제) → 카테고리별 클러스터링. media_count≥2 필터로 의제설정이론 적용
  • 데이터 계약: interfaces.py에 LLMResponse·ToolContext·ToolLoopResult 3개 dataclass로 모듈 간 인터페이스 정의. mypy strict으로 컴파일 타임 검증
  • 선택적 노출 모델: 동일 뉴스 풀에서 LLM이 자율 선택 → 선택적 노출(Stroud, 2011)의 자연 발생 관찰

기술적 도전과 해결

뉴스가 투표에 영향을 미치지 않는 문제

v1에서 뉴스를 프롬프트에 직접 주입 → 투표 결과 불변 (선행연구와 동일 실패). 원인: LLM이 주입된 정보를 "경험"이 아닌 단순 "정보"로 인식. → Function Calling 자율 탐색으로 전면 재설계하여 해결

OpenAI vs Gemini FC 포맷 비호환

두 API의 메시지 구조가 근본적으로 상이. LangChain 래퍼는 FC 세부 제어를 제한. → llm_client.py(278줄) 통합 래퍼 직접 구현, 상위 모듈에는 동일한 LLMResponse 인터페이스만 노출

2,000명 규모 비용 최적화

초기 추정 시뮬레이션 1회 $400+. → Gemini Flash Lite($0.10/1M), 여론조사 200명 표본추출, FC 루프 단일 세션 통합으로 ~$50으로 87% 절감

E-Commerce QA Auto-Evaluation Pipeline

2026.02 ~ 2026.04

이커머스 상품 Q&A 데이터를 크롤링·수집하고, 최대 7개 LLM이 독립 평가 → 다수결 투표 → 소수파 설득 토론을 순차 수행하는 3-Phase 자동 품질 판정 파이프라인. 다수결의 신뢰성은 독립 판단자 수에 비례한다는 원칙으로 98.8% 교차 일치율 달성.

왜 멀티 프로바이더인가

서로 다른 학습 데이터·아키텍처의 모델을 조합하여 특정 모델 편향이 결과를 지배하지 않도록 설계. Groq 무료 티어 + GPT-5 mini 조합으로 건당 비용 최소화.

트리플 OCR

상품 상세 이미지의 텍스트 추출에 단일 OCR은 한국어 인식률 불안정. EasyOCR(범용) + PaddleOCR(아시아어) + Surya(레이아웃 인식) 3중 교차 검증으로 신뢰도 확보.

봇 감지 우회

Playwright + Stealth v2 + Chrome 쿠키 자동 복사 + 홈페이지 워밍업 3단계 전략. Akamai CDN의 CDP 탐지를 우회.

핵심 구현

  • 3-Phase Debate Engine: Phase 1(N개 모델 병렬 독립 평가, ThreadPoolExecutor) → Phase 2(기준별 다수결, 과반 확정/min() 보수적 채택) → Phase 3(소수파 설득 토론 2라운드, 합의 실패 시 다수결 채택)
  • 팩토리+세션 패턴: AgentFactory(ABC)로 OpenAI/Groq/OpenRouter/Anthropic 통일. 세션 기반 대화(Phase 1~2)와 단발 호출(Phase 3 토론) 분리
  • Q 4기준 통합 호출: 기준별 개별 호출 시 4배 비용. 통합 프롬프트로 기준 간 일관성 향상 + Pydantic 검증
  • 토론 스킵 최적화: 같은 그룹에 확정 0점이 있으면 min() 사전 시뮬레이션으로 불필요한 토론 생략

기술적 도전과 해결

SDP Truncation 오분류

relevant_SDP_text 최대 22,000자가 8,000자로 절단 → 핵심 정보 누락으로 Answerability 오판. Evidence와 Answerability의 교차 검증 패턴으로 발견. → SDP 전체를 프롬프트에 임베딩, 모델별 context window에 동적 조절

Akamai 봇 감지 우회

CDP 연결 자체를 탐지하여 UA 변경·webdriver 플래그만으로 우회 불가. → Playwright + stealth v2 + Chrome 쿠키 복사 + 홈페이지 워밍업 3단계. 배치 크롤링(5개 단위) + 쿨다운(5~8분)으로 IP 차단 방지

모델 간 경계 사례 불일치

3자 비교에서 27건 불일치, 대부분 "partially_answerable" 판단. SOP 자체의 모호성이 원인. → 사람 전건 리뷰로 해석 기준 확립, 자동화 범위를 "기준이 명확한 항목"으로 한정

CCXT(표준화 조회) + python-binance(선물 특화 실행) 이원화. "조회는 표준화, 실행은 특화" 원칙. 거래소 변경 시 CCXT만 교체.

4-상태 머신

NO_POSITION → ENTERING → HAS_POSITION → USER_MANAGED. "거래소가 진실" 원칙 — 내부 상태 불일치 시 거래소 기준 자동 보정. 17가지 엣지케이스 명세서로 정의.

학술 검증

77개월(5개 시장 레짐) 실증으로 규칙 기반 DCA의 구조적 안정성 증명. "시그널이 틀려도 시스템이 무너지지 않는" 실패 흡수 구조를 학술적으로 검증.

핵심 구현

  • 명세서 기반 설계: 7개 명세서(전략/진입/청산/개입/에러/알림/아키텍처)와 코드의 1:1 대응. 17가지 사용자 개입 시나리오(수동 TP/SL, 부분 체결, 강제 청산 등) 전체 명세화
  • "거래소가 진실" 원칙: 내부 상태와 거래소 불일치 시 거래소 기준으로 자동 보정. StateManager가 JSON으로 영속화하여 봇 재시작 시에도 상태 유지
  • 시그널봇 895줄 통합 엔진: BB·RSI·StochRSI·CCI·VRSI 7개 지표 × 8 타임프��임 × 3모드. shift(1) 일관 적용으로 look-ahead bias 완전 방지
  • 다이버전스 자동 탐지: RSI/거래량 다이버전스를 lookback + skip_recent 파라미터로 자동 감지. 3중 중복 방지(캔들 시간 + 쿨다운 + 포지션 ID)
  • 백테스트 Look-ahead 방지: 모든 지표에 shift(1), 진입은 bar open 기준. 환상적 백테스트 성과를 구조적으로 차단

상태 동기화 문제
